## 1. 调试与观测

开发前必须优先寻找：

- 游戏日志。
- Mod 加载器日志。
- 崩溃日志。
- 启动参数。
- 配置校验结果。
- 系统事件日志。

没有日志时，必须使用替代方式：

- 文件差异对比。
- 二分法排查。
- 最小复现。
- 单变量修改。
- 人工验证清单。

出现崩溃时，必须先记录：

- 崩溃时间。
- 崩溃场景。
- 最近一次修改。
- 是否稳定复现。
- 回滚后是否恢复。

## 2. 开发循环

单点开发循环固定为：

1. 确认本次只改一个目标点。
2. 记录修改目的和预期结果。
3. 修改文件。
4. 执行最小验证。
5. 记录实际结果。
6. 通过后进入下一目标点。
7. 失败时进入问题修复闭环。

多个修改必须拆成多个单点循环。

禁止把多个未验证修改一次性合并。

## 3. 问题修复闭环

出现问题时，必须先确认 root cause，再修改。

禁止只根据上一次修改反复试错。

每次修复必须记录映射：

```md
## 修复记录

版本：

原问题：

当前修改：

新问题：

root cause：

验证方式：

结论：
```

修复顺序固定为：

1. 复现问题。
2. 读取相关文件和上下文。
3. 确认最近修改。
4. 分析 root cause。
5. 制定最小修改。
6. 执行验证。
7. 更新知识库。
