## 1. 测试规则

强交互类型游戏使用人工验证清单。

开发过程中，能自动化的流程必须自动化。

必须优先自动化：

- 游戏启动。
- 游戏关闭。
- 游戏重启。
- Mod 安装。
- Mod 卸载。

必须优先执行：

- 脚本语法检查。
- 文件存在性检查。
- 安装脚本检查。
- 卸载脚本检查。
- 打包内容检查。
- 人工进游戏验证。
- 发布包反装验证。

自动化流程失败时，必须记录失败原因。

无法自动化的验证项，必须加入人工验证清单。

自动化失败项未完成替代验证前，不得声明测试完成。

每次测试前，AI 必须提供测试列表。

测试列表固定格式：

```md
## 测试列表

- [ ] 安装脚本 `--dry-run`。
- [ ] 安装 Mod。
- [ ] 启动游戏。
- [ ] 验证 Mod 是否生效。
- [ ] 进入相关场景。
- [ ] 关闭游戏。
- [ ] 重启游戏。
- [ ] 卸载脚本 `--dry-run`。
- [ ] 卸载 Mod。
- [ ] 启动游戏确认恢复原状。
- [ ] 从发布包重新安装验证。
- [ ] 记录测试结果。
```

人工验证清单至少包含：

- 游戏能否启动。
- Mod 是否生效。
- 进入相关场景是否崩溃。
- 退出游戏是否正常。
- 卸载后游戏是否恢复原状。

## 2. 交付目录和发布包规则

发布包固定命名：

```text
{Mod名称}-{版本号}.zip
```

发布包名称禁止包含本机绝对路径。

`package/` 是当前版本最终交付目录。

发布包必须从 `package/` 生成，并在验证通过后放入 `releases/`。

`package/` 和发布包必须包含：

- Mod 文件。
- 安装脚本。
- 卸载脚本。
- `manifest.json`。

发布包反装验证固定流程：

1. 将发布包解压到临时目录。
2. 检查发布包内文件清单。
3. 使用发布包内安装脚本按 `manifest.json` 安装。
4. 启动游戏或执行替代验证。
5. 使用发布包内卸载脚本卸载。
6. 验证回滚结果。
7. 记录验证结论。

发布包未完成反装验证前，不得放入 `releases/`。

## 3. 交付规则

交付前必须确认：

- 本版本目标已完成或明确放弃。
- Git 当前变更已检查。
- `docs/status.md` 已更新。
- 开发文档已更新。
- 安装脚本能执行。
- 卸载脚本能执行。
- 安装脚本 `--dry-run` 能执行。
- 卸载脚本 `--dry-run` 能执行。
- 发布包命名符合固定格式。
- 发布包已完成反装验证。
- 已检查安装目标文件冲突。
- 安装成功提示为中文并包含版本号。
- 卸载成功提示为中文并包含版本号。
- `manifest.json` 已更新。
- 知识库已更新。
- 已检查是否需要更新 `AGENTS.md`。
- 已完成测试列表中的验证。
- 发布包已放入 `releases/`。
- Git commit 状态已明确：已按用户确认提交，或用户未确认提交且已记录未提交状态。

未满足以上条件，不得声明版本完成。
