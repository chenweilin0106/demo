# 游戏 Mod 开发流程规范

## 1. 文档定位

本文档用于约束 AI 开发游戏 Mod 时的工作流程。

目标：

- 减少盲猜。
- 先确认合规边界、环境基线和验收标准。
- 保证每次修改能追踪。
- 保证安装、卸载、回滚能执行。
- 保证每个版本完成后更新知识库。
- 保证每个版本完成后检查项目规则。

本文档不是 Mod 原理科普，不记录泛泛的逆向成本说明。

## 2. 术语定义

- 开发目录：存放 Mod 开发文件、文档、知识库、脚本、Git 仓库。
- 游戏根目录：Steam 或本地安装的游戏主目录，用于识别、测试和人工验证。
- 安装目标目录：`manifest.json` 中 `installTarget` 指向的绝对路径，是安装脚本实际写入 Mod 文件的目录。
- 发布包：从 `package/` 生成并放入 `releases/` 的 zip 文件。
- 安装记录：安装脚本写入安装目标目录的记录文件，用于卸载和回滚。
- 版本记录：当前版本的变更摘要，固定写入 `docs/{版本号}.md`。
- 开发期备份：开发过程中手动或脚本生成的临时备份。
- 安装期备份：安装脚本覆盖文件前生成的备份。

开发目录必须放在游戏根目录外。

游戏根目录不作为安装脚本参数，不作为脚本安装路径来源。

开发目录、游戏根目录、安装目标目录必须在开发文档中明确记录。

## 3. 固定流程

流程分为两类：

- 项目初始化：新建 Mod 项目时执行一次。
- 版本开发：每个版本都必须执行。

项目初始化完成前，禁止开始功能开发。

每个版本必须按以下顺序执行：

1. 读取上下文。
2. 确认合规边界。
3. 确认环境基线。
4. 确认版本目标和验收标准。
5. 收集资料。
6. 判断 Mod 类型。
7. 选择工具。
8. 制定安装、卸载、回滚方案。
9. 执行最小验证。
10. 单点开发循环。
11. 执行安装、卸载、回滚验证。
12. 生成测试列表。
13. 执行自动化验证和人工验证。
14. 打包发布包。
15. 从发布包重新安装验证。
16. 更新开发文档。
17. 更新知识库。
18. 检查 `AGENTS.md`。
19. 更新版本记录。
20. 检查 Git 变更。
21. 如用户明确确认提交，则创建 Git commit。
22. 交付版本。

禁止跳过读取上下文、合规边界、环境基线、验收标准、安装卸载回滚、测试列表、发布包反装验证、知识库更新、`AGENTS.md` 检查。

## 4. 核心原则

- 先读上下文，再修改。
- 先做最小验证，再做完整功能。
- 先保证安装和回滚，再发布。
- 先记录结论，再进入下一轮猜测。
- 开发文档记录目标和结论，Git 记录文件变化。
