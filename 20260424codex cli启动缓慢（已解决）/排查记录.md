# Codex CLI 启动缓慢排查记录

记录时间：2026-04-24 21:08 CST

## 目标

尽量缩短在 WSL Ubuntu 终端中输入 `codex` 后，到 Codex CLI 主界面可交互之间的等待时间。

优先级：

1. 官方支持的配置项或环境变量。
2. 简单、可回滚、不破坏核心功能的关闭项。
3. 明确定位后再考虑 wrapper、降级、替换终端或其他规避方案。

## 当前现象

- 当前截图显示 Codex CLI 版本：`0.124.0`。
- 当前模型/模式：`gpt-5.5 xhigh`。
- 用户观察：输入 `codex` 后，到截图中的启动弹框/界面出现，约等待 10 秒。
- 历史对比：用户记得 `0.116.0` 版本启动是毫秒级；`0.117.0` 之后开始变慢。

## 已验证或已尝试

这些信息来自用户描述，后续如有必要再重新取证。

- 发现过启动阶段存在多个 `403` 请求。
- 已尝试关闭部分 feature：

```toml
[features]
apps = false
plugins = false
tool_suggest = false
```

- 关闭上述 feature 后，效果不明显。
- 曾怀疑 WSL 访问 Windows 侧配置导致慢。
- 已尝试在 WSL 内放置配置，并将 Codex home 目录改到 WSL 侧，效果不明显。
- 曾怀疑与终端光标定位或渲染有关，尚未形成结论。

## 排查原则

- 先复现并测量，不直接猜测修复。
- 每次只改变一个变量。
- 优先使用只读命令和临时环境变量。
- 不永久修改 `~/.codex/config.toml`、shell 启动文件、认证文件或安装版本；如需修改，先单独确认。
- 记录每个测试的命令、耗时、现象和结论。

## 主要假设

### H1：启动阶段等待网络请求、权限检查或失败重试

依据：

- 用户之前观察到多个 `403`。
- `apps`、`plugins`、`tool_suggest` 可能会触发远端能力、插件或建议相关请求。

待验证：

- `codex --help`、`codex --version` 是否也慢。
- 干净 `CODEX_HOME` 是否仍慢。
- 临时禁用代理变量、改变网络环境、抓取 debug 日志后，是否能看到阻塞请求。

### H2：启动阶段读取了较慢路径或大量本地配置

依据：

- WSL 访问 Windows 文件系统通常比 Linux 原生路径慢。
- Codex 可能扫描配置、MCP、插件、技能、历史记录或 auth 文件。

待验证：

- `which codex` 和实际入口位置。
- `CODEX_HOME`、`HOME`、相关配置文件位置。
- `strace` 是否显示大量 `/mnt/c/...` 文件访问或慢系统调用。

### H3：终端渲染、光标定位或 TUI 初始化慢

依据：

- 截图中的等待发生在 TUI 初始界面出现前后。
- 如果 `--help`、`--version` 快，而交互 TUI 慢，则更偏向该方向。

待验证：

- 非交互命令和交互命令的耗时差异。
- 不同 `TERM`、`NO_COLOR`、终端宽度或 Windows Terminal 设置是否影响启动。

### H4：版本变更引入启动期新能力初始化

依据：

- 用户观察 `0.117.0` 后开始变慢。

待验证：

- 查看本机安装方式与版本。
- 如果可以安全访问官方资料，确认是否有启动配置项、feature flag 或已知变更。
- 不先降级；降级只作为最后对照实验。

观察：

- 这解释了 `script -t` 中看到的多个约 2.18 秒台阶。

- `ESC[6n` 光标位置读取对应 Codex 日志里的 warning，是已确认的一个超时点。
- `OSC 10;?` 颜色查询没有在 Codex 日志中形成明显 warning，但从输出序列和 crossterm 源码看，它也可能贡献等待。

### T11：主动响应 PTY 终端查询后的对照

状态：已完成第一轮。

测试目的：

- 验证“如果终端查询都能立即收到响应，启动是否能恢复到毫秒级”。
- 这可以区分：
  - 纯终端探测超时；
  - 终端探测之外的 session 初始化等待。

测试方法：

- 使用 `pty.fork()` 启动 `codex --no-alt-screen`。
- 监视子 PTY 输出。
- 一旦看到终端查询，就主动回包：
  - `ESC[6n` -> `ESC[1;1R`
  - `ESC[c` -> `ESC[?1;0c`
  - `OSC 10;?` -> `OSC 10;rgb:eeee/eeee/eeee ST`
  - `OSC 11;?` -> `OSC 11;rgb:0000/0000/0000 ST`

结果摘要：

```text
responses at ~0.605s: cursor CPR, primary DA, OSC10, OSC11
first_text_seconds ~= 6.907
first_big_seconds ~= 6.907
```

观察：

- 终端查询已经在约 0.6 秒内被主动响应。
- 首次大规模绘制仍然约 6.9 秒才出现。
- 这说明终端探测超时能解释一部分等待，但不能解释全部启动慢。
- 当前 10 秒级等待更像是两个因素叠加：
  - 终端查询响应不完整或超时，贡献约 3 秒以上；
  - session 初始化早期存在约 5 秒空档。

## 当前结论

阶段性结论：

- 当前 `codex --version` 和 `codex --help` 约 0.03 秒，说明 Node/npm 入口、二进制加载、基础配置解析不是瓶颈。
- 当前交互 TUI 启动的 10 秒级等待，不是单一原因。
- 第一部分是“初始光标位置读取/终端能力探测超时”。
- 源码层面已经确认至少三类 2000ms 级别的终端查询会参与启动：
  - `ESC[6n`：光标位置读取；
  - `ESC[?u` / `ESC[c`：keyboard enhancement 支持探测；
  - `OSC 10;?` / `OSC 11;?`：默认终端颜色查询。
- `CODEX_TUI_DISABLE_KEYBOARD_ENHANCEMENT=1` 可以跳过其中一类探测，本轮测试约节省 3 秒左右。
- 第二部分是 session 初始化早期的约 5 秒空档。
- 主动响应 `ESC[6n`、`ESC[c`、`OSC 10;?`、`OSC 11;?` 后，首屏仍约 6.9 秒才出现，说明终端探测不是全部原因。
- 这个约 5 秒空档出现在 `Codex::spawn_internal` 中、`ExecPolicyManager::load(...)` 之前。
- 目前源码和配置状态已经排除或降低了 plugins、Node 兼容解析、MCP、`auth_mcp` 的优先级。
- `skills.include_instructions=false` 不是有效启动加速项。
- `project_doc_max_bytes=0` 不是有效启动加速项，`AGENTS.md` 读取不是当前主因。
- 临时 no-skills 环境能把 `thread_spawn -> exec_policy` 从约 6.5 秒降到约 4.1-5.0 秒，说明 skills 扫描/安装检查贡献一部分成本，但仍有约 4 秒未解释。
- 当前没有证据表明主因是：
  - MCP server 启动超时；
  - `auth_mcp` 15 秒等待；
  - `shell_snapshot`；
  - 旧的 plugin/app 相关 `403`；
  - 单独的 Windows 侧 Codex home 符号链接。
- 官方支持的配置项中，目前没有找到能直接跳过 `get_cursor_position()` 或缩短该等待的开关。
- 当前也没有找到能通过 config 关闭默认颜色查询的配置项。
- 下一步最值得验证的是剩余约 4 秒空档到底落在哪些系统调用或子进程上。

## 当前可行建议

优先保留：

```toml
[features]
apps = false
plugins = false
tool_suggest = false
```

如果未来配置了 MCP server，再按需使用官方支持项：

```toml
[mcp_servers.<name>]
enabled = false
# 或仅降低失败等待
startup_timeout_sec = 1
```

不建议作为核心启动加速方案：

- `TERM=dumb`：已验证无效。
- `--no-alt-screen`：可保留滚屏，但不解决核心等待。
- 精简 `status_line`：不解决核心等待。
- `tui.terminal_title=[]`：可减少标题设置，但不解决核心等待。
- 将 `CODEX_HOME` 移到 WSL：可减少部分 I/O 风险，但不是本轮 10 秒等待主因。

当前最值得实测的临时启动方式：

```bash
CODEX_TUI_DISABLE_KEYBOARD_ENHANCEMENT=1 codex
```

说明：

- 这是临时环境变量，不会修改配置文件。
- 预期只能减少一部分等待，不能完全恢复到毫秒级。
- 需要在真实 Windows Terminal 窗口中手动体感确认，因为 `script` 子 PTY 不等同于外层真实终端。

可以继续评估、但暂不建议直接持久化的启动方式：

```bash
CODEX_TUI_DISABLE_KEYBOARD_ENHANCEMENT=1 codex -c 'skills.bundled.enabled=false'
```

说明：

- 这个组合可能继续减少一部分启动成本。
- 但 `skills.bundled.enabled=false` 会影响内置 skills，并可能删除临时/真实 `CODEX_HOME/skills/.system` 下的 system skills。
- 在真实配置中启用前，需要明确接受这个取舍。

下一步方向：

1. 在真实终端中手动运行 `CODEX_TUI_DISABLE_KEYBOARD_ENHANCEMENT=1 codex`，确认体感是否明显改善。
2. 在真实终端中临时运行 `CODEX_TUI_DISABLE_KEYBOARD_ENHANCEMENT=1 codex -c 'skills.bundled.enabled=false'`，仅做体感对照，不先写入配置。
3. 用另一个真实终端环境对照，例如 Windows Terminal 以外的终端，确认是否是 Windows Terminal/WSL PTY 对 `ESC[6n`、`OSC 10;?` 等查询响应不完整或响应路径异常。
4. 如需要更强证据，可安装或使用 `strace` 后抓一次交互启动系统调用，重点定位 `thread_spawn -> exec_policy` 之前的 4-6 秒。
5. 向上游提交 issue 时，附上以下关键信息：
   - `codex-cli 0.124.0`
   - WSL Ubuntu，`TERM=xterm-256color`，`WT_SESSION` 存在。
   - 非交互命令 0.03 秒。
   - `script -t` 显示约 12 秒才开始绘制界面。
   - 日志反复出现 `failed to read initial cursor position; defaulting to origin`。
   - `CODEX_TUI_DISABLE_KEYBOARD_ENHANCEMENT=1` 能减少一部分等待，但仍会卡在 `ESC[6n` 和 `OSC 10;?` 相关探测。
   - `session_init.mcp_manager_init enabled_mcp_server_count=0`。
