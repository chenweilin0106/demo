# winget 踩坑笔记

## 1. `winget` 能运行，但某些窗口里报 “系统无法访问此文件”

现象（示例）：
- 运行 `winget source update` / `winget upgrade` 时提示无法启动进程，路径指向：
  - `C:\Users\<你>\AppData\Local\Microsoft\WindowsApps\winget.exe`

常见原因：
- `winget.exe` 通过 “App Execution Alias” 转发到 App Installer，某些终端/权限组合下启动失败（尤其是管理员窗口/策略限制/别名失效）。

处理思路（从轻到重）：
1) 换一个新开的终端窗口再试（很多时候就好了）。
2) 在管理员 PowerShell 里跑一次：
   ```powershell
   winget --version
   ```
   如果管理员窗口能用而普通窗口不能用（或反过来），优先固定你常用的一种方式。
3) 确认系统里有 **App Installer**（Microsoft.DesktopAppInstaller）。没有的话需要从 Microsoft Store 安装/修复。

## 2. `winget upgrade` 显示 “找不到可用的升级”，但软件自己弹窗提示有更新

原因可能是：
- 软件的“应用内更新”发布比 `winget` 源更快，短期不同步。
- 软件走“增量补丁更新”（delta patch）容易失败，导致“先卸载后补丁失败”，看起来像被卸载。

建议：
- 先 `winget source update` 再看 `winget upgrade`。
- 对更新容易翻车的软件（例如 Docker Desktop 这类带自更新器的），优先用 **官方全量安装器** 升级，而不是点应用内的增量更新按钮。

## 3. `source update` / `install` 网络失败（代理/公司网络）

排查：
```powershell
winget source list
winget source update
```

如果你在代理环境：
- `winget` 更依赖系统代理（WinHTTP/系统代理），可检查：
  ```powershell
  netsh winhttp show proxy
  ```
- 不同网络环境差异很大：能直连就别强行配复杂代理；如果必须代理，建议按公司 IT 的 WinHTTP 代理规范配置。

## 4. `msstore` 源相关问题

现象：
- 搜不到软件、安装失败、提示需要登录/地区限制等。

建议：
- 优先用 `winget` 社区源安装常见软件。
- 需要 Store 的包再指定来源：
  ```powershell
  winget install --source msstore <query>
  ```

## 5. 安装后命令找不到（PATH 没刷新）

现象：
- `winget install ...` 成功，但新开命令报 “not recognized”。

原因：
- 安装器写了 PATH（用户/系统），但当前终端不会自动刷新。

处理：
1) 关闭并重开终端窗口。
2) 如果仍不行，检查 PATH 里是否有目标目录（用户 PATH/系统 PATH）。

## 6. Docker Desktop 更新“把自己卸载了”的复盘（本机案例）

关键信息：
- Docker Desktop 走了增量补丁更新（下载包很小），先卸载旧版，再对 `C:\Program Files\Docker\Docker` 打补丁。
- 补丁失败后回滚不完整，导致 `docker`、服务、注册项缺失，看起来像“消失/卸载”。

更稳的升级策略：
- 避免点击应用内增量更新按钮；
- 用 `winget` 或官方全量安装器升级（全量覆盖安装更抗失败）。
