# jsonImage 开发文档

## 背景

现有 `svgaImage` 已经把 SVGA 动画加载与缓存收敛到独立组件中：组件负责播放状态与生命周期，缓存模块负责 IndexedDB 读写、同 URL 并发去重和失败降级。

现有 `titleImg` 主要服务称号动画，支持静态图、普通 lottie json 和 zip lottie 包。其中 zip 包通过 IndexedDB 缓存原包，再在渲染时解压、读取 `data.json`，并把 zip 内图片资源临时转成 data URI 后交给 `lottie.loadAnimation`。

`jsonImage` 的目标是把 `titleImg` 中和“称号”无关的 lottie 动画能力抽出来，做成通用 JSON 动画组件。它只聚焦 lottie json/zip 动画，静态图继续交给现有图片组件处理。

## 目标

- 新增 `jsonImage` 目录，后续包含 `jsonImage.vue`、`jsonZipIdb.js` 等自包含文件。
- 组件只处理 lottie 动画资源：普通 json、目录形式 json、zip 包。
- 缓存设计与 `svgaImage` 的整体思路保持一致：组件不直接关心 IndexedDB 细节，缓存模块提供统一获取方法。
- 缓存对象与 SVGA 不同：`svgaImage` 缓存 `videoItem`，`jsonImage` 缓存 zip 原包 `ArrayBuffer`。
- 不缓存 base64，也不缓存解压后的 `animationData`。
- IndexedDB 不可用、读取失败或写入失败时自动降级，不影响动画渲染。

## 不做事项

- 不处理静态图片。
- 不改动现有 `titleImg`。
- 不抽取 `svgaImage` 与 `jsonImage` 的公共 IndexedDB helper。
- 不做 TTL、容量淘汰或主动清理策略，除非业务后续明确需要。
- 不把 zip 解压后的图片 base64 写入 IndexedDB。

## 对外 API

### Props

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `imgName` | `String` | `''` | 动画资源名、相对路径或完整 URL |
| `loop` | `Boolean` | `true` | 是否循环播放 |

### Events

| 事件 | 触发时机 |
| --- | --- |
| `DOMLoaded` | lottie 触发 `DOMLoaded` 时透传 |
| `complete` | 非循环播放完成时触发 |
| `loaded` | 动画实例创建并完成首轮可渲染准备后触发 |
| `error` | json/zip 加载、解压或解析失败时触发 |

`DOMLoaded` 与 `complete` 用于兼容 `titleImg` 的现有使用习惯；`loaded` 和 `error` 用于和 `svgaImage` 的使用体验靠近。

## 资源识别规则

`jsonImage` 不再判断图片格式，只区分 zip 与 json。

- 以 `.zip` 结尾：按 zip lottie 包处理。
- 以 `.json` 结尾：按普通 lottie json 文件处理。
- 其它字符串：按目录处理，补全为 `data.json` 和 `images/`。

判断后缀时需要兼容 query/hash，例如：

- `xxx.zip?v=1`
- `xxx/data.json?v=1`

## 路径规则

保留 `titleImg` 中已有的路径思路：

- 完整 `http(s)` URL 直接使用。
- 非完整 URL 走 `process.env.VUE_APP_OSS_PATH` 拼接。
- 目录路径补全：
  - `path`: `${dir}/data.json`
  - `assetsPath`: `${dir}/images/`
- json 文件路径补全：
  - `path`: 原始 json 地址
  - `assetsPath`: json 同级目录下的 `images/`

实现时建议把路径处理集中在组件内部的纯函数中，避免散落在加载流程里。

## 缓存模块设计

文件建议命名为 `jsonZipIdb.js`。

### 数据库结构

```js
const DB_NAME = 'json_image_zip_db'
const DB_VERSION = 1
const STORE_NAME = 'json_zip_list'
```

记录结构：

```js
{
  url: string,
  zipBuffer: ArrayBuffer,
  updatedAt: number
}
```

`url` 使用完整资源 URL，作为 objectStore 的 `keyPath`。

### 对外方法

```js
export function getJsonZipData(url)
```

职责：

1. 参数为空时返回 `Promise.resolve(null)`。
2. 同 URL 命中 `inflight Map` 时复用同一个 Promise。
3. 打开 IndexedDB。
4. 先读缓存，命中 `zipBuffer` 后直接返回。
5. 未命中时用 `axios.get(url, { responseType: 'arraybuffer' })` 拉取 zip 原包。
6. 异步写入 IndexedDB，不阻塞渲染。
7. 写入失败时静默降级。
8. Promise settle 后清理 `inflight`。

### 降级策略

- `window.indexedDB` 不存在：直接走网络请求。
- open 失败：直接走网络请求。
- get 失败：忽略缓存，继续网络请求。
- put 失败：忽略写入结果，不影响当前动画。
- 网络请求失败：抛给组件，由组件触发 `error`。

## 为什么缓存 zip 原包

zip 原包通常比把图片转成 base64 后的内容更小。

base64 会带来额外体积膨胀，如果把转换后的 data URI 或完整 `animationData` 写入 IndexedDB，会增加存储占用，也更容易触发空间不足。当前更合理的策略是：

- IndexedDB 只缓存 zip 原包。
- 渲染时临时解压 zip。
- 只在本次渲染内把 zip 内图片转成 data URI。
- 动画实例销毁后，相关临时数据交给浏览器回收。

这个策略的取舍是：缓存命中后能省掉网络下载，但仍需要本地解压和 base64 转换。相比缓存完整 `animationData`，它更节省存储空间，也更接近资源原始形态。

## 组件渲染流程

### 初始化与切换

`imgName` 变化时：

1. `loadVersion += 1`。
2. 销毁旧 lottie 实例。
3. 清空旧状态。
4. 根据资源类型进入 json 或 zip 加载流程。

所有异步回调都必须校验当前版本：

```js
if (currentVersion !== this.loadVersion) return
```

这样可以避免旧资源的异步回调覆盖新资源。

### 普通 json

1. 规范化资源地址。
2. 调用 `lottie.loadAnimation`。
3. 传入：
   - `container`
   - `renderer: 'svg'`
   - `loop`
   - `autoplay: true`
   - `path`
   - `assetsPath`
4. 绑定 `DOMLoaded`、`complete`、错误相关事件。

普通 json 不额外做 IndexedDB 缓存，主要依赖浏览器 HTTP 缓存。

### zip 包

1. 生成完整 zip URL。
2. 调用 `getJsonZipData(zipUrl)` 获取 zip 原包。
3. 使用 `JSZip.loadAsync(zipBuffer)` 解压。
4. 查找 `data.json`：
   - 优先查根目录 `data.json`。
   - 找不到时在所有 zip 文件里匹配 `(^|/)data.json`。
5. 解析 `animationData`。
6. 遍历 `animationData.assets`：
   - 跳过空 asset。
   - 跳过 `asset.e === 1` 的外链或已内嵌资源。
   - 根据 `asset.u + asset.p`、`asset.p`、去掉 `images/` 前缀后的路径查找 zip 内图片。
   - 找到后读取 base64，设置：
     - `asset.u = ''`
     - `asset.p = data URI`
     - `asset.e = 1`
7. 调用 `lottie.loadAnimation({ animationData })`。

## 生命周期

- `beforeDestroy`：销毁 lottie 实例，递增 `loadVersion`。
- 如果组件后续需要配合 `keep-alive`，可以再补充 `activated/deactivated`：
  - `activated` 恢复播放。
  - `deactivated` 暂停播放。

首版可以先保持和 `titleImg` 一致，只处理销毁，不额外引入播放状态管理。

## 错误处理

组件内部需要捕获以下错误：

- zip 下载失败。
- zip 解压失败。
- zip 中缺少 `data.json`。
- `data.json` 不是合法 JSON。
- lottie 容器不存在。
- lottie 实例创建失败。

错误处理原则：

- 控制台保留必要错误日志，便于排查资源问题。
- 对外触发 `error`，让业务按需展示兜底 UI。
- 不在组件内部处理静态兜底图，因为静态图不属于 `jsonImage` 的职责。

## 与 svgaImage 的统一点

- 都由组件处理生命周期、实例销毁、事件透传。
- 都由独立缓存模块处理 IndexedDB。
- 都用完整 URL 作为缓存 key。
- 都用 `inflight Map` 合并同 URL 并发请求。
- 都在 IndexedDB 不可用或写入失败时自动降级。
- 都不在缓存模块里做业务 UI 逻辑。

## 与 svgaImage 的差异点

| 项目 | `svgaImage` | `jsonImage` |
| --- | --- | --- |
| 动画引擎 | `svgaplayerweb` | `lottie-web` |
| 缓存对象 | `videoItem` | zip 原包 `ArrayBuffer` |
| 缓存收益 | 跳过 SVGA 解压/解码 | 跳过 zip 网络下载 |
| 命中后是否仍需解析 | 通常不需要重新 `Parser.load` | 仍需 `JSZip.loadAsync` 和图片 data URI 转换 |
| 普通资源加载 | SVGA 都走 parser | 普通 json 走 lottie 原生 `path/assetsPath` |

## 从 titleImg 迁移时注意

- 当前 `titleImg` 的 `import { getTitleZipData } from '@/views/m4/titleZipIdb'` 带有具体业务路径，`jsonImage` 应改为目录内自包含依赖。
- `TitleImg` 的组件名、class 名、错误文案里有称号语义，`jsonImage` 需要改成通用语义。
- 静态图片分支不要迁移进来。
- `getZipAnimationData`、`getDataJsonFile`、`getZipAssetFile`、`getMimeType` 这些 zip 处理逻辑可以复用思路，但需要改名为通用命名。
- 保留 `loadVersion` 防乱序机制。

## 建议验收点

- 传入普通 `.json` URL 可以正常播放。
- 传入目录地址可以自动加载 `data.json` 和 `images/`。
- 传入 `.zip` URL 首次能下载并播放。
- 第二次加载同一个 zip URL 能命中 IndexedDB，Network 不再重复下载 zip。
- 同一个 zip URL 并发渲染时只发起一次下载。
- 快速切换 `imgName` 时旧动画不会覆盖新动画。
- zip 中 `data.json` 不在根目录时仍能找到。
- zip 内图片路径带 `images/` 或不带 `images/` 都能匹配。
- IndexedDB 禁用或写入失败时仍能播放，只是不缓存。
- 非循环播放完成后能触发 `complete`。
- 组件销毁后 lottie 实例被销毁，不继续播放。
