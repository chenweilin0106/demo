# 榜单奖励更新流程

本文档用于指导 Codex 根据奖励截图、页面组件、弹框组件和参考模板，更新 `components/rank` 活动的榜单奖励数据、页面展示、规则弹框和预览弹框。

执行前必须先读取本文档，整理输入与修改计划，并等待用户确认；确认前不得新增、删除或修改任何项目文件。

## 调用方式

每次任务开始时，明确要求 Codex 读取并严格按照本文档执行：

```text
请先读取并严格按照以下文档执行：
/home/dcw220/projects/demo/components/rank/榜单奖励更新流程.md
```

## 必要输入

开始修改前，先确认这些信息：

| 项目 | 要求 |
| --- | --- |
| 活动目录 | 当前活动所在目录或组件路径。 |
| 页面组件 | 需要更新页面奖励展示的 Vue 文件。 |
| 弹框组件 | 需要更新规则弹框和预览弹框展示的 Vue 文件；必须以页面组件实际 import 为准确认路径，解析不到实际文件时先停止并反馈。 |
| 图片目录 | 当前活动图片资源目录；用户提供时，资源检查优先只在该目录和活动目录内进行。 |
| 奖励截图 | 日榜、总榜、奖励名称、数量、天数、价格角标、新奖励角标。 |
| 弹框标题图片 | 每个榜单的日榜标题图、总榜标题图的图片名称或路径。 |
| `rankChosen` 对应关系 | 例如 `1` 为人气榜、`2` 为神豪榜、`3` 为家族榜。 |
| 页面运行地址 | 默认 `http://localhost:8080/#/rank`；用户给完整地址时以用户输入为准。 |
| 特殊说明 | 例如图片不需要处理、资源 `404` 可忽略。 |

参考模板默认用于字段格式对照：

- 页面模板：`/home/dcw220/projects/demo/components/rank/index.vue`
- 弹框模板：`/home/dcw220/projects/demo/components/rank/popups/rulesPopup.vue`，不能替代弹框组件实际路径确认。
- 预览弹框模板：`/home/dcw220/projects/demo/components/rank/popups/previewPopup.vue`，不能替代弹框组件实际路径确认。

如果用户提供了更贴近当前活动的参考模板路径，以用户提供的路径为准。缺少页面组件、弹框组件或 `rankChosen` 对应关系时，先问用户确认；除非组件代码里已经能明确核实，不要凭记忆猜。

## 修改前整理

修改前先把本次任务整理成一份简短计划，至少覆盖：

| 检查项 | 需要说明的内容 |
| --- | --- |
| 数据量变化 | 原始页面榜单数量、目标页面榜单数量、原始弹框榜单数量、目标弹框榜单数量、是否需要扩展模板结构。 |
| 多榜单映射 | 榜单名称、`rankChosen`、日榜页面 key、总榜页面 key、弹框奖励列表名。 |
| 页面奖励矩阵 | 按 `rankChosen × tagChosen` 列出所有页面奖励 key、每个 key 的目标奖励项数量、是否存在占位项。 |
| 日榜奖励 | 榜单名称、排名、奖励名称、数量或天数、`mark`、是否新奖励、图片或字段名。 |
| 总榜奖励 | 榜单名称、排名或分组、奖励名称、数量或天数、`mark`、是否新奖励、图片或字段名。 |
| 预览字段 | 哪些奖励需要预览，以及对应的 `type`、`previewImg`、`previewTip1`。 |
| 图片目录 | 用户提供的图片目录、活动目录内资源目录、资源检查范围。 |
| 礼物价格说明 | 日榜表格下方、总榜奖励下方或其他位置的说明文案。 |
| 弹框标题图片 | 每个榜单的日榜标题 class、总榜标题 class、背景图路径、资源实际宽高、资源检查结果和占位处理。 |
| 弹框布局 | `nSp` 阈值、最后一排是否需要 `noMargin`、总榜奖励分组数量、是否需要滚动到底部。 |

计划里必须写清楚拟修改文件、榜单覆盖范围、数据结构变化、预览字段变化和待确认问题。用户确认后再改文件。

## 修改范围

默认只改这些内容：

- 奖励数据。
- 标题图。
- 必要榜单映射。
- 必要说明文案。
- 已确认影响展示的布局字段。
- 奖励项字段中的 `type`、`previewImg`、`previewTip1`、`mark`、`isNew`。

除非用户明确要求，不改表格 DOM 结构、原有 class、整体样式和无关布局。不自行启动项目；用户会先启动项目。如果运行地址打不开，只反馈页面不可访问并请用户确认服务状态。

## 榜单数据规则

- 目标榜单数量以用户截图和说明为准，不以原始组件已有数组数量为准。
- 如果原始只有 1 个榜单，但目标需要 3 个榜单，不要把 3 个榜单混进同一份奖励数组。
- 页面组件按 `rankChosen` 和日榜/总榜维度补齐对应奖励数据 key；key 形态以当前组件实际逻辑为准。
- 页面组件使用 ``rewardsList[`${rankChosen}${tagChosen}`]`` 这类动态 key 时，必须先列出完整 `rankChosen × tagChosen` 页面奖励矩阵，例如 `11`、`12`、`21`、`22`。
- 每个页面奖励 key 都必须写明目标奖励项数量，并逐项核对奖励名称、数量或天数、`mark`、`isNew`、`type`、`icon`、`previewImg`、`previewTip1`。
- 页面 `rewardsList` 中不得保留空的 `type`、`icon`、`text` 占位项；确实无法确认时，必须在计划和最终输出中标为待确认。
- 弹框奖励完整不代表页面奖励完整；必须以页面渲染表达式实际读取的 key 为准单独检查。
- 弹框组件按榜单新增成组结构：标题图、日榜表格、日榜说明、总榜标题、总榜奖励列表。
- 弹框奖励列表按现有命名延续，例如 `rewardsList`、`rewardsList2`、`rewardsList3`。
- 如果原组件没有支持多榜单展示，需要同步扩展模板结构、数据结构和样式。
- `rankChosen` 映射必须来自用户输入或组件代码，不能猜测 `1`、`2`、`3` 分别是什么榜。
- `tagChosen` 默认只处理日榜和总榜；除非当前组件代码另有特殊含义，否则不要要求用户额外填写。

## 截图识别和资源确认

- 截图中的奖励名称、数量、天数、价格角标和新角标必须二次核对；小字、压缩图或相近字形导致低置信度时，先标记为待确认，不自行定名。
- 奖励名称不能只靠 OCR 结果确定，必须结合用户说明、页面模板、弹框模板和当前组件上下文核对。
- 资源确认优先使用用户提供的图片目录；未提供时，只检查活动目录内常规资源目录和当前组件已有资源引用。
- 禁止为了找资源做全仓大范围搜索；找不到资源时记录缺失，不继续扩大范围。
- 图片、json、svga 等资源 `404` 或缺失不能用于反推奖励名称，只记录为资源异常。

## 奖励项字段规则

更新奖励项时，不能只改 `icon`、`text` 和 `mark`。逐项核对这些字段：

| 字段 | 规则 |
| --- | --- |
| `type` | 奖励类型，同时影响预览弹框样式。填写预览字段时必须同步确认。 |
| `icon` | 奖励图标字段；页面组件按页面模板格式更新，弹框组件按弹框模板格式更新，不能把两边格式混用。 |
| `text` | 页面展示的奖励名称或奖励说明。 |
| `mark` | 通常表示奖励图片上的价格角标，例如 `17999`、`6666`。 |
| `previewImg` | 预览弹框使用的图片或动效资源；只有它有值时页面才会显示预览入口。 |
| `previewTip1` | 预览弹框第一行提示文案。填写 `previewImg` 时必须同步填写或确认。 |
| `mark1` | 以当前组件含义为准，常用于额外角标或特殊展示，不要擅自改语义。 |
| `isNew` | 新奖励角标；必须根据截图或用户说明确认，不要自行推断。 |

价格已经放入 `mark` 时，不要再把同一价格重复写进奖励底部文案。礼物价格说明通常放在奖励表格或奖励分组下方，不放进单个奖励项文案。

页面组件和弹框组件字段格式不完全相同。更新时必须分别对照页面模板和弹框模板中同类型奖励的字段写法，只替换本次奖励对应的名称、数量、天数、价格、新角标和资源名；模板没有的字段不要自行补结构，模板已有字段不要遗漏。

页面奖励项点击后通常通过 `previewImgCk(item)` 打开预览弹框，并把 `type`、`previewImg`、`previewTip1` 传入弹框。模板中常见需要预览或已有专门样式的类型包括 `pst_privilege`、`car`、`title`、`CP_card`、`world_msg_skins`、`room_dec_suit`、`costume`、`ring`。`chat_bubble`、`chat_bubble_pub`、`voice`、`mic`、`tool` 等类型通常不填预览字段；但如果本次活动截图、用户说明或当前组件已有数据给出 `previewImg`，必须保留或同步更新。

## 页面和弹框同步

- 页面组件和弹框组件需要同步处理新角标，不要只改其中一处。
- 页面组件和弹框组件使用同一批奖励数据时，预览字段必须跟随页面奖励项一起更新。
- 规则弹框如果单独维护奖励列表，也要检查是否存在相同奖励项字段或展示入口。
- 日榜奖励表格只放排名和奖励；日榜下方如有礼物价格说明，作为表格后的说明文案。
- 总榜奖励一般放在独立标题下，以分组奖励列表展示。
- 大部分活动有两个或更多榜单，不能只更新一个榜单数据。

## 弹框布局规则

- `reward` 控制奖励项展示。
- `nSp` 控制榜单奖励分组的额外间距。
- `item.id < x ? 'nSp' : null` 中的 `x` 表示从哪一项开始不加 `nSp`。
- 如果 6 个总榜奖励分组且最后 1 项单独展示，阈值通常是 `5`，不是 `6`。
- 如果最后一排有两个奖励分组，可参考模板注释使用 `noMargin` 逻辑；最后只有一个奖励分组时通常不需要 `noMargin`。
- 新增多个榜单后，需要重新检查弹框滚动区域是否能展示完整内容。

## 弹框标题图片规则

- 每个榜单通常有两个标题图：日榜标题和总榜标题。
- 标题图一般用 `div` 元素和背景图实现。
- class 按榜单顺序递增：第一个榜单 `img1`、`img2`，第二个榜单 `img3`、`img4`，第三个榜单 `img5`、`img6`。
- 标题图资源必须使用用户提供的图片名称或路径；未提供时，先问用户确认。
- 新增标题图时，宽高以该标题图资源的实际尺寸为准；只参考当前 `img1`、`img2` 的 margin 居中和背景图写法。
- 标题图资源必须确认存在且可正常加载；不存在或加载异常时，先保留占位样式继续流程，任务结束时统一反馈异常资源。
- 背景图目录以当前页面组件或弹框组件实际写法为准，常见写法：
  - `background: url('@/assets/tk_51.png') no-repeat left top/100% 100%;`
  - `background: url('@/pages/childrenDay/assets/tk_51.png') no-repeat left top/100% 100%;`
- 新增 `img3`、`img4` 等 class 时，同时检查 `<template>` 和 `<style scoped>`，避免元素存在但背景不显示。

## 自检要求

- 除非用户明确要求，不运行 `npm run build:test`。
- 使用用户提供或默认运行地址做页面运行检查，不追加额外 query 或 hash 参数。
- 使用 `$agent-browser` 基于用户已启动的页面做实时调试，包括打开页面、截图或快照、点击弹框、滚动检查内容。
- 如需验证不同榜单或日榜/总榜状态，优先通过修改默认数据或默认选中状态测试，测试后必须还原；临时测试改动也必须写入确认计划；不通过地址栏参数绕过页面逻辑。
- 图片、json、svga 等资源 `404` 或加载失败可忽略并记录，不继续排查。
- 需要记录并处理白屏、弹框打不开、数据不展示、预览打不开、脚本异常等非资源加载类问题。

- 日榜、总榜和多榜单映射是否都已覆盖。

- 新角标、`mark`、`type`、`previewImg`、`previewTip1`、标题图片宽高和资源状态、弹框布局是否已检查。
- 页面、规则弹框、预览弹框的自检方式与结果。
- 已忽略的资源加载问题。
- 无法确认的截图内容、字段含义或运行时问题。
